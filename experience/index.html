# Experience

지금까지 수행한 주요 프로젝트와 그 과정에서 해결한 주요 문제, 사용한 기술, 그리고 기여 내용을 정리했습니다.

# 와이펫 (2023 - 현재)

# 1. 반려동물 건강관리 B2C 플랫폼 개발 (와이펫)

  • 기술 스택: NestJS(TypeScript), Flutter, MongoDB, Redis, AWS(ECS, Amplify, S3, CloudWatch), GitHub Actions
  • 역할: 백엔드 아키텍처 설계 및 서비스 전반 개발 주도

# 문제

  • 블루투스 기반 체온 측정 장치로부터 수집되는 데이터를 안정적으로 처리하고, 개체별로 상이한 정상 체온 범위를 반영한 개인화된 건강 관리 기능이 필요했으며, 서비스 확장을 고려한 구조적 설계가 요구되는 상황이었음.
  • 기존 서비스는 아키텍처가 현대적이지 않아 유지보수 비용이 높았고, 기능의 안정성과 신뢰성이 충분히 확보되지 않은 상태였음.
  • 동일한 기능을 제공하는 두 개의 프로젝트로 분리된 구조로 인해 서비스가 중복 운영되고 있었으며, 이를 단일 서비스로 통합하는 과정과 함께 데이터베이스 마이그레이션이 필요한 상황이었음.
  • 기존에 이용하던 클라우드 서비스가 불안정하여 배포를 위한 새로운 인프라를 결정해야 하는 상황이었음

# 해결 / 기여

  • 기존 개발자들과의 면담을 통해 기술 스택 및 개발 환경에 대한 요구사항을 수집하고, 이를 반영하여 프로젝트 초기 아키텍처를 설계함.
  • 분리되어 운영되던 다수의 서비스와 프로젝트를 효율적으로 통합하고 일관된 구조로 관리하기 위해 모노레포 기반 구조를 채택함.
  • 한 명의 프론트엔드 개발자가 웹, iOS, Android를 모두 담당해야 하는 환경을 고려하여, 모바일 앱은 Flutter 기반으로 통합 개발할 수 있도록 구성함.
  • NestJS 기반 모노레포 아키텍처를 설계하고, 메인 서비스, 디바이스 연동 서비스, 공통 라이브러리로 모듈을 분리하여 유지보수성과 확장성을 확보함.
  • 블루투스 스캐너로부터 수집된 데이터를 처리하는 백엔드 API를 설계하고, 데이터 수집 → 저장 → 조회까지의 전체 데이터 흐름을 정의 및 구현함.
  • 개체별 평소 체온 데이터를 기반으로 개인화된 기준 체온을 산출하는 로직을 설계 및 구현하여, 이상 탐지 및 건강 상태 판단 기능을 고도화함.
  • 기존 서비스 간 데이터 구조 차이를 분석하고, 통합 과정에서 데이터 정합성을 유지하기 위한 DB 마이그레이션 전략을 수립 및 수행함.
  • 안정적인 서비스 운영을 위해 AWS 기반 인프라(ECS, S3, CloudWatch 등)를 구성하고, CI/CD 환경을 GitHub Actions 기반으로 구축하여 배포 자동화를 구현함.

# 결과

  • 디바이스 연동 → 데이터 수집 → 처리 → 사용자 서비스로 이어지는 End-to-End 백엔드 시스템을 구축함.
  • 서비스 및 프로젝트 구조를 단일 모노레포로 통합하여 개발 생산성과 유지보수 효율성을 개선함.
  • 개인화된 체온 분석 기능을 통해 기존 단순 수집형 서비스에서 데이터 기반 헬스케어 서비스로의 차별화를 달성함.
  • 서비스 구조 및 핵심 기능에 대한 기술 특허 진행으로 기술적 차별성과 확장 가능성을 확보함.
  • 확장 가능한 아키텍처 및 인프라를 기반으로 향후 기능 추가 및 서비스 확장에 대응 가능한 구조를 마련

# 2. AWS 인프라 구조 개선 및 비용 최적화

  • 기술 스택: AWS (ECS, S3, CloudWatch), Docker
  • 역할: 인프라 설계 및 운영

# 문제

  • 초기 인프라 구성에서 리소스가 비효율적으로 사용되고 있어 운영 비용이 증가하고 있었으며, 서비스 확장에 대응하기 위한 구조 개선이 필요한 상황이었음.
  • 배포 프로세스가 체계적으로 정립되어 있지 않아, 장애 대응 및 hotfix 상황에서 신속하고 안정적인 배포가 어려웠음.

# 해결 / 기여

  • 서비스 구조 및 리소스 사용 현황을 분석하여 불필요한 리소스를 식별하고 정리함.
  • ECS 기반으로 서비스 배포 구조를 재정비하고, 컨테이너 기반 운영 환경을 표준화함.
  • S3 및 CloudWatch를 활용하여 로그 및 스토리지 관리 구조를 개선하고, 운영 가시성을 확보함.
  • 서비스 특성과 트래픽 패턴을 고려하여 적절한 리소스 할당 및 스케일링 전략을 적용함으로써 비용 대비 성능을 최적화함.
  • GitHub Actions 기반 CI/CD 파이프라인을 구축하여 PR 병합 시 자동 빌드 및 배포가 수행되도록 구성함.
  • 배포 과정에서 장애 발생 시 즉시 이전 버전으로 복구할 수 있도록 롤백 가능한 배포 구조를 설계 및 적용함.

# 결과

  • AWS 크레딧 약 $5,750 범위 내에서 안정적인 인프라 운영을 유지하며 비용 최적화를 달성함.
  • 배포 자동화 및 롤백 체계를 구축하여 배포 안정성과 대응 속도를 개선함.
  • 리소스 사용 효율을 개선하고 운영 프로세스를 표준화함으로써 지속 가능한 인프라 운영 구조를 확보함.

# 3. 오프라인 매장 시스템 및 온라인 연동 구조 설계 (펫스테이션)

  • 기술 스택: NestJS, Next.js, POS/키오스크 연동 시스템
  • 역할: 시스템 아키텍처 설계 및 데이터 파이프라인 구축

# 문제

  • 오프라인 매장을 새로 오픈함에 따라 발생하는 주문 및 결제 데이터를 온라인 서비스와 통합 관리할 수 있는 구조가 부재하여, 데이터 단절 및 운영 비효율이 발생하고 있었음.

# 해결

  • POS, 키오스크, 무인 주문 시스템 등 다양한 결제 및 주문 채널을 연동하는 구조를 설계함
  • 매장 운영 데이터 및 정산 데이터를 수집·관리하는 파이프라인을 구축함
  • 오프라인 매장과 온라인 플랫폼 간 데이터 동기화 구조를 설계하여 통합 운영 환경을 구현함
  • 쿠폰, 이벤트, QR 기반 리워드 시스템 등 사용자 참여를 유도하는 핵심 기능을 설계 및 개발함

# 결과

  • 오프라인 매장 시스템 구축 및 안정적 운영 (2024년 11월 오픈)
  • 온·오프라인 데이터 통합을 통한 운영 효율성 향상

# 4. 개발팀 구축 및 기술 조직 운영

  • 역할: 개발팀 리딩 및 프로세스 개선

# 문제

  • 초기 조직에서 개발 프로세스와 협업 방식이 정립되지 않아, 개발 생산성과 커뮤니케이션 효율이 낮은 상태였음.

# 해결

  • 스크럼 기반 개발 프로세스를 도입하여 작업 단위 및 일정 관리 체계를 구축함
  • OKR 기반 목표 관리 시스템을 운영하여 조직 단위의 방향성을 명확히 함
  • 주간 기술 공유 세션(금요 인사이트)을 운영하여 팀 내 기술 역량을 지속적으로 향상시킴
  • AWS 및 외부 벤더사와의 협업을 통해 기술적 의사결정 지원

# 결과

  • 개발 프로세스 표준화 및 협업 효율 개선
  • 팀 단위 기술 성장 및 지식 공유 문화 정착

# 월간해킹 (2021 - 2023)

# 보안 취약점 점검 플랫폼 및 관련 서비스 백엔드 개발

  • 기술 스택: NestJS (TypeScript), Express, Monorepo, Bull, RabbitMQ, MariaDB, MongoDB, Redis, AWS, Sequelize
  • 역할: 백엔드 아키텍처 설계 및 서비스 전반 개발

# 문제

  • 보안 취약점 점검, 피싱 메일 훈련, 버그 바운티 등 다양한 보안 서비스가 요구되었으나, 이를 통합적으로 운영하고 확장할 수 있는 구조가 부재한 상황이었음.
  • 취약점 검사와 같이 실행 시간이 길고 자원이 많이 소모되는 작업을 안정적으로 처리할 수 있는 비동기 처리 및 작업 관리 구조가 필요했음.
  • 서비스별로 데이터 성격과 처리 방식이 달라 단일 데이터베이스 구조로는 효율적인 관리 및 확장이 어려운 상황이었음.
  • 고객사 및 사용자 유형이 다양하여 역할 기반 접근 제어 및 서비스 분리가 필요한 환경이었음.

# 해결 / 기여

  • NestJS 기반 Monorepo 아키텍처를 설계하여 사용자, 취약점 검사, 워커, 리포트 생성, 관리자 서비스 등을 도메인 단위로 분리하고 통합 관리할 수 있는 구조를 구축함.
  • Bull Queue 및 RabbitMQ를 활용하여 취약점 검사 및 리포트 생성과 같은 장시간 작업을 비동기 이벤트 기반으로 처리하는 시스템을 설계 및 구현함.
  • Worker 프로세스를 분리하여 API 서버와 작업 처리 로직을 분리하고, 작업 실행의 신뢰성과 확장성을 확보함.
  • MariaDB, MongoDB, Redis를 데이터 특성에 따라 분리하여 사용하고, 로그, 캐시, 트랜잭션 데이터를 각각 적절히 관리할 수 있도록 구조를 설계함.
  • 취약점 검사 요청부터 결과 저장, 리포트 생성까지 이어지는 데이터 흐름을 정의하고, 대용량 결과 데이터를 효율적으로 처리할 수 있는 구조를 구현함.
  • Toss Payments를 연동하여 결제 및 서비스 이용 흐름을 구축하고, 사용자 및 관리자 역할에 따른 권한 기반 접근 제어를 구현함.
  • 프론트엔드 팀과 협업하여 API 스펙을 공유하고, 서비스 요구사항에 맞는 인터페이스를 설계 및 조정함.

# 결과

  • 취약점 검사, 피싱 메일 훈련, 버그 바운티 플랫폼 등 서로 다른 성격의 보안 서비스를 하나의 Monorepo 기반 플랫폼으로 통합함.
  • 비동기 큐 기반 아키텍처를 도입하여 장시간 작업의 안정적인 처리와 시스템 신뢰성을 확보함.
  • 도메인 단위로 분리된 서비스 구조를 통해 유지보수성과 확장성을 개선하고, 다양한 보안 서비스 요구사항에 대응 가능한 백엔드 플랫폼을 구축함.
  • 결제, 사용자 관리, 취약점 검사, 리포트 생성 등 핵심 기능을 포함한 보안 서비스 플랫폼의 기반 아키텍처를 설계 및 구현함.

# 유이수 (2019 - 2020)

# 디어유 및 유통/배송 관리 시스템 개발 및 인프라 개선

  • 기술 스택: Node.js (Express), Python (Flask), PHP, Sequelize, MySQL, MariaDB, AWS (EC2, Lambda, API Gateway, Route 53, VPC), Docker, Zapier, Git
  • 역할: 백엔드 개발 및 시스템 구조 개선 주도

# 문제

  • 기존 서비스들이 서로 다른 구조와 기술 스택으로 구성되어 있어 유지보수 및 확장성이 낮고, 시스템 간 통합이 어려운 상황이었음.
  • 유통/배송/주문 데이터를 관리하는 시스템에서 외부 주문 데이터를 수집하고 통합하는 자동화 구조가 부재하여 수작업 의존도가 높았음.
  • 서비스 트래픽 증가에 따라 서버 및 네트워크 구조를 재설계하고 보안 및 안정성을 개선할 필요가 있었음.
  • 여러 프로젝트가 분리된 상태에서 데이터 일관성과 관리 효율성이 부족한 문제가 존재했음.

# 해결 / 기여

  • 기존 시스템 구조를 분석하고, 서비스 간 역할 분리를 고려한 개선된 아키텍처를 설계 및 적용함.
  • Express 기반 백엔드와 React 프론트엔드를 분리하여 프론트/백엔드 구조를 명확히 분리하고 유지보수성을 개선함.
  • Sequelize ORM 및 마이그레이션 기반 데이터베이스 관리 구조를 도입하여 스키마 변경 및 데이터 관리의 안정성을 확보함.
  • Docker 기반 개발 환경을 구성하여 로컬 개발 및 배포 환경을 표준화하고 협업 효율을 향상시킴.
  • 유통/배송 관리 시스템에서 주문, 재고, 판매 데이터를 통합 관리할 수 있는 데이터 구조를 설계 및 구현함.
  • Zapier와 AWS Lambda를 활용하여 외부 주문 데이터를 자동으로 수집 및 처리하는 서버리스 기반 데이터 파이프라인을 구축함.
  • AWS EC2, API Gateway, Route 53, VPC 등을 활용하여 네트워크 및 인프라 구조를 재구성하고, 트래픽 대응 및 보안 수준을 개선함.
  • SSL 적용 및 네트워크 구조 개선을 통해 서비스 안정성과 보안성을 강화함.
  • 미니앱 관리 시스템 및 위챗 기반 서비스의 유지보수 및 기능 개선을 수행함.

# 결과

  • 유통/배송/주문 데이터를 통합 관리할 수 있는 시스템을 구축하여 운영 효율성을 개선함.
  • 서버리스 기반 자동화 구조(Zapier + Lambda)를 통해 외부 데이터 수집 프로세스를 자동화하고 수작업 의존도를 감소시킴.
  • Docker 기반 개발 환경 및 프론트/백엔드 분리를 통해 협업 구조를 표준화하고 개발 생산성을 향상시킴.
  • AWS 기반 인프라 재구성을 통해 서비스 안정성과 확장성을 확보함.
  • 다양한 프로젝트를 하나의 일관된 구조로 개선하는 과정에서 시스템 아키텍처 및 인프라 전반에 대한 실무 경험을 축적함.

# 개인 프로젝트

# MCP 기반 AI 페르소나 기억 관리 시스템 (agent-persona-system)

  • 기술 스택: TypeScript, Node.js, Prisma, PostgreSQL (pgvector), MCP SDK, Gemini Embedding API
  • 역할: 시스템 설계 및 MCP 서버 구현, RAG 기반 메모리 관리 구조 설계

# 문제

  • LLM 기반 에이전트는 세션 간 컨텍스트가 유지되지 않아, 대화가 길어질수록 맥락이 일관되게 유지되지 않는 문제가 있었음.
  • 장기적인 작업 흐름에서 이전 대화나 중요 정보를 효과적으로 참조하지 못해, 일관된 의사결정 및 응답 생성에 한계가 존재했음.
  • 단순 프롬프트 기반 접근만으로는 지속적인 기억 관리 및 맥락 유지가 어려워, 외부 메모리 시스템과의 연동이 필요한 상황이었음.

# 해결 / 기여

  • RAG(Retrieval-Augmented Generation) 패턴을 기반으로, 에이전트가 외부 기억을 검색하고 활용할 수 있는 메모리 시스템을 설계 및 구현함.
  • Gemini Embedding API를 활용하여 텍스트를 768차원 벡터로 변환하고, PostgreSQL의 pgvector를 이용해 벡터 기반 저장 및 검색 구조를 구축함.
  • 쿼리 임베딩과 저장된 벡터 간 코사인 유사도(<=> 연산자)를 활용하여 의미적으로 유사한 기억을 검색하도록 구현함.
  • 벡터 유사도 검색과 함께 중요도(1–5), 카테고리, 생성 시점 등의 메타데이터 필터링을 결합하여 검색 정밀도를 향상시킴.
  • 세션 시작 시 loadOn: session_start 메타데이터를 기반으로 필수 기억을 자동 로드하여 초기 컨텍스트를 구성하도록 설계함.
  • 세션 중단 및 복구 기능을 구현하여, INTERRUPTED 상태의 세션을 감지하고 이전 작업 내용을 요약 및 복구할 수 있도록 지원함.
  • 기억 간 중요도 기반 분류 및 링크 구조를 설계하여, 개별 기억 간 관계를 표현하고 탐색 가능하도록 구현함.
  • MCP SDK를 활용하여 단순 API 연동이 아닌 MCP 서버 자체를 직접 구현하고, Claude Code와 연동하여 에이전트가 지속적으로 페르소나 상태를 유지하도록 구성함.
  • Prisma와 PostgreSQL 기반으로 데이터 모델을 설계하여 기억, 메타데이터, 관계 정보를 구조적으로 관리함.

# 결과

  • LLM 에이전트가 세션을 넘어 일관된 페르소나와 맥락을 유지할 수 있는 메모리 시스템을 구축함.
  • RAG 기반 구조를 통해 필요한 기억을 동적으로 검색·활용할 수 있는 아키텍처를 설계함.
  • 벡터 검색 + 메타데이터 필터링을 결합하여 검색 정확도와 컨텍스트 활용도를 개선함.
  • MCP 서버 형태로 구현하여 실제 개인 워크플로우에서 지속적으로 활용 가능한 형태의 AI 시스템을 완성함.
